# 备份与恢复建议

本文用于上线前制定最低可执行的备份与恢复方案。当前项目还没有自动备份流水线，因此上线前需要先建立人工备份习惯。

## 需要备份的内容

- Supabase Postgres 数据库
- Supabase Storage `public-media`
- Supabase Storage `private-media`
- 生产 `.env` 中的真实配置
- 项目代码仓库

生产 `.env` 必须安全离线保存，不要放入公开仓库、聊天记录、截图或前端构建产物。

## 数据库备份

建议方式：

- 使用 Supabase 控制台的数据库备份能力。
- 或使用 `pg_dump` 导出 PostgreSQL 数据。

建议频率：

- 上线初期：每次大量写入、迁移、媒体整理前手动备份。
- 稳定期：至少每周备份一次。
- 重要内容发布后：额外做一次手动备份。

备份前建议记录：

- 备份时间
- Supabase 项目 ref
- 当前代码 commit
- 是否包含迁移后数据
- 是否包含媒体元数据

## Storage 备份

需要分别备份：

- `public-media`
- `private-media`

建议：

- 定期导出或同步 bucket 文件。
- 保留原 bucket 名称和路径结构。
- 恢复时不要把 private 文件放入 public bucket。
- 不要因为备份方便把 `private-media` 改成公开。

## 恢复流程

1. 拉取项目代码仓库。
2. 配置生产环境变量。
3. 恢复 Supabase Postgres 数据库。
4. 恢复 `public-media` bucket 文件。
5. 恢复 `private-media` bucket 文件，并确认 bucket 仍然非公开。
6. 运行项目校验命令。
7. 验证管理员登录。
8. 验证公开文章访问。
9. 验证 private-media 只在管理员授权视图下显示。

建议恢复后运行：

```bash
npm run check
npm test
npm run build
npm run verify:supabase
```

## 恢复后人工验证

- 首页能打开。
- 一篇公开文章能打开。
- 管理员能登录 `/admin`。
- 普通用户不能访问 `/admin`。
- 管理员能看到私密 / 草稿 / 归档文章 badge。
- `public-media` 图片正常显示。
- `private-media` 图片只在管理员视图显示。
- 站点设置能读取，不白屏。

## 安全提醒

- 不要把 `SUPABASE_SERVICE_ROLE_KEY` 放入公开仓库。
- 不要把生产 `.env` 写入 README 或文档。
- 不要把 `private-media` 改成公开 bucket。
- 不要把 signed URL 写入数据库。
- 不要把临时 signed URL 保存到 `posts.markdown`。
- 默认管理员密码必须在生产上线前修改。
