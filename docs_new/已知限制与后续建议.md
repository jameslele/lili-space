# 已知限制与后续建议

本文记录上线前已知限制、当前不做的能力，以及后续可以优先考虑的优化方向。

## 访问延迟

当前数据和媒体仍使用 Supabase。Supabase 项目节点位于韩国，国内访问时可能出现数据库查询、Storage 图片、音视频加载延迟。

短期建议：

- 保持当前 Supabase 架构。
- 继续减少前台重复查询。
- 优先优化首页和文章页图片加载。

中长期建议：

- 如果国内访问仍然慢，优先考虑把媒体迁到阿里云 OSS + CDN。
- 如果数据库 TTFB 长期偏高，再评估迁移到阿里云 RDS PostgreSQL 或其他更近区域的 PostgreSQL。

## 构建提示

`npm run build` 可能出现 Vite chunk size warning。

当前判断：

- 主要来源是后台文章编辑器 `PostEditor` / Milkdown Crepe。
- 前台公开页面不会加载后台编辑器重型 JS。
- 不阻塞上线。

后续建议：

- 仅在后台编辑页继续保持页面级加载。
- 不为了消除 warning 重写编辑器。
- 如后台编辑器首载明显变慢，再考虑更细的动态拆包。

## 图片和媒体

当前限制：

- 图片未接入专门压缩流水线。
- 未接入 CDN 图片处理。
- 未生成响应式缩略图。
- 音频和视频只做上传与原生播放。
- 不做转码、视频封面抽帧、码率处理。

后续建议：

- 优先为公开图片接入 CDN。
- 评估迁移媒体到阿里云 OSS。
- 对首页封面和 Gallery 图片建立缩略图或压缩版本。

## 评论、点赞和搜索

当前未实现：

- 评论
- 点赞
- 搜索

普通用户账号系统已经存在，但当前主要是为未来评论、点赞等能力预留。上线时不要在界面或文档里承诺这些功能已经可用。

## 备份自动化

当前未实现完整自动备份流水线。

上线前至少需要人工确认：

- Supabase Postgres 可恢复。
- `public-media` 和 `private-media` 有可导出的备份方式。
- 生产 `.env` 离线安全保存。

后续建议：

- 建立定期数据库备份。
- 建立 Storage bucket 定期同步或导出。
- 对恢复流程做一次演练。

## private-media signed URL

private-media 必须保持非公开。

当前边界：

- signed URL 只由服务端生成。
- 只有管理员授权视图会获得 private-media signed URL。
- 普通用户和访客不能获得 private-media signed URL。
- signed URL 有效期当前为 1 小时。
- signed URL 不写入数据库。

注意：

- `posts.markdown` 中不要保存临时 signed URL。
- `media_assets.public_url` 可以保存 Supabase 生成的对象 URL，但 private-media bucket 不应公开读取。
- 不要为了前台显示方便把 `private-media` 改成公开 bucket。

## Vercel 预览

当前项目使用 `@astrojs/vercel`。

限制：

- `astro preview` 不支持 Vercel adapter。
- `npm run preview` 会失败，这是 adapter 限制，不代表构建失败。

本地 Vercel 预览建议使用：

```bash
npx vercel dev
```
