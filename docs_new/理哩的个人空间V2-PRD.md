# 理哩的个人空间 V2 PRD

Version: 2.0
Date: 2026-06-12

---

# 一、修订背景
本版基于 `docs` 目录下 V1 系列文档，并参考 `lilisong` 目录中的 Hexo 博客项目重新整理。

核心调整：

+ 不再把文章设计成 Travel、Essays、Journal、Moments 这类固定内容类型。
+ 改为接近 `lilisong` 的文章组织方式：统一文章主体 + 分类 + 标签 + 归档。
+ 登录不再采用 Supabase Auth 邮箱验证码登录，改为项目自建账号密码登录。
+ 数据存储继续使用 Supabase Postgres，媒体文件继续使用 Supabase Storage。
+ 管理员初始账号为 `root`，初始密码为 `1212`。
+ 后台编辑器从 Tiptap / Notion-like 块编辑器改为 Milkdown Crepe。

---

# 二、产品定位
理哩的个人空间是一个私人博客与生活记录网站。

它更像：

+ 一个长期更新的个人博客
+ 一个安静的文字与照片空间
+ 一个可按分类、标签、时间回看生活的私人档案

它不是：

+ 内容社区
+ 商业 CMS
+ 技术博客模板
+ 重运营型自媒体站点

---

# 三、技术选型
V2 推荐技术栈：

| 模块 | 选型 | 说明 |
| --- | --- | --- |
| 前台框架 | Astro | 适合内容驱动网站，页面以服务端渲染和静态内容体验为主 |
| UI 交互 | React Islands | 仅在后台、登录、编辑器等需要交互的位置使用 |
| 数据库 | Supabase Postgres | 继续作为主数据存储，不使用 Supabase Auth |
| 文件存储 | Supabase Storage | 存放文章图片、封面图、Gallery 图片等媒体资源 |
| 鉴权 | 自建账号密码登录 | 使用 `users`、`sessions` 等业务表，不接入 Supabase Auth |
| 密码处理 | bcrypt / Argon2 | 数据库只保存密码哈希，不保存明文 |
| 编辑器 | Milkdown Crepe | 后台 Markdown 编辑器，正文源数据保存为 Markdown |
| Markdown 渲染 | remark / rehype | 前台将 Markdown 渲染为 HTML，并做必要的安全处理 |
| 样式 | Tailwind CSS | 使用 Tailwind 构建页面样式，少量 CSS Variables 承载颜色、字体等设计 token |

选型依据：

+ Astro 官方定位适合博客、文档、作品集等内容驱动网站。
+ Supabase 项目提供完整 Postgres 数据库，适合作为文章、用户、分类、标签等结构化数据存储。
+ Supabase Storage 适合存储和分发图片、视频等用户生成媒体。
+ Milkdown Crepe 与 Markdown 写作模型匹配，比私有富文本结构更利于迁移。
+ Tailwind CSS 适合在 Astro 项目中快速落地统一间距、排版、响应式和组件状态。

参考资料：

+ Astro Docs: https://docs.astro.build/
+ Supabase Database Docs: https://supabase.com/docs/guides/database/overview
+ Supabase Storage Docs: https://supabase.com/docs/guides/storage
+ Milkdown Crepe Docs: https://milkdown.dev/docs/guide/using-crepe

# 四、目标用户
## 1. 管理员
理哩本人。

核心需求：

+ 写文章
+ 上传图片
+ 编辑封面、摘要、分类、标签
+ 发布或保存草稿
+ 管理历史文章
+ 调整站点基础信息
+ 在前台查看全部文章，包括仅自己可见文章

## 2. 注册用户
已注册并登录的普通用户。

V2 中普通用户的核心价值是为未来评论、点赞、互动功能预留账号体系。

当前版本能力：

+ 登录账号
+ 浏览公开文章
+ 浏览公开分类、标签、归档、照片和关于页面

限制：

+ 不能进入后台
+ 不能查看仅自己可见文章
+ 暂不提供评论和点赞功能

## 3. 访客
理哩的朋友或偶然访问者。

核心需求：

+ 阅读公开文章
+ 通过分类浏览文章
+ 通过标签发现相关内容
+ 通过归档按年份回看内容
+ 浏览照片与关于页面

# 五、内容模型
## 1. 统一文章
V2 只有一种核心内容：

```plain
post
```

所有长文、短文、旅行记录、日常片段、读书观影记录、技术备忘都作为文章存储。

## 2. 分类
分类用于表达文章所属的主要栏目。

参考 `lilisong` 中已有分类风格，分类可以是更个人化、文学化的命名，而不是产品化类型。

可选初始分类：

```plain
游游逛逛
卿卿我我
少侠
浮光片影
但是还有书籍
四季流转
花花草草
网站日记
码农小知识
未分类
```

说明：

+ 一篇文章建议只选择一个主分类。
+ 分类允许后台新增、重命名、隐藏。
+ 分类名称面向读者展示，可以保留个人表达。

## 3. 标签
标签用于表达文章主题、地点、对象或关键词。

示例：

```plain
北京
城市
游记
推荐
书影音
夏
技术
```

说明：

+ 一篇文章可以有多个标签。
+ 标签不承担导航主结构，只用于补充关联。

## 4. 归档
归档按照发布时间自动生成。

推荐维度：

+ 年份归档
+ 月份归档

## 5. 可见性
文章支持两种可见性：

```plain
public
private
```

说明：

+ `public`：公开文章，访客、普通登录用户、管理员都能看到。
+ `private`：仅自己可见文章，只有管理员登录后能在前台和后台看到。

文章状态和可见性是两个独立维度：

+ `status` 控制文章是否处于草稿、已发布、归档。
+ `visibility` 控制谁可以阅读文章。

# 七、登录与权限
V2 实现账号密码登录，但不使用 Supabase Auth。

用户角色：

```plain
admin
reader
```

说明：

+ `admin`：理哩本人，可以进入后台，可以在前台看到全部文章。
+ `reader`：普通注册用户，当前只能浏览公开文章，为未来评论、点赞预留。

## 1. 管理员初始账号

初始账号：

```plain
root
```

初始密码：

```plain
1212
```

## 2. 普通用户注册登录
V2 支持普通用户注册和登录。

注册字段：

```plain
用户名
密码
显示名，可选
```

登录字段：

```plain
用户名
密码
```

V2 不要求邮箱验证码，不要求手机号，不要求第三方登录。

## 3. 前台文章可见性规则
+ 未登录访客：只能看到 `status = published` 且 `visibility = public` 的文章。
+ 普通登录用户：只能看到 `status = published` 且 `visibility = public` 的文章。
+ 管理员登录用户：前台可以看到全部文章，包括 `private` 文章；草稿和归档文章需要显示状态标识，避免误以为已公开。

## 4. 后台权限规则
+ 只有 `role = admin` 可以访问 `/admin`。
+ 普通用户访问 `/admin` 时跳转到前台或显示无权限页面。
+ 后台文章列表可以管理全部文章。

实现原则：

+ `/admin` 及其子页面需要登录。
+ 不使用 Supabase Auth。
+ 不提供好友权限申请。
+ V2 暂不设计好友分组、单篇授权、申请访问等复杂权限。

安全说明：

+ `root / 1212` 仅作为开发初设账号密码。
+ 数据库不得明文保存密码，应保存密码哈希。
+ 后续上线前应修改初始密码。

# 八、非目标
V2 暂不做：

+ Supabase Auth 邮箱登录
+ 好友可见内容
+ 单篇文章访问申请
+ 评论系统
+ 点赞系统
+ 搜索引擎优化专项
+ 多作者协作
+ 复杂工作流审批

# 九、成功标准
+ 管理员可以用 `root / 1212` 登录后台。
+ 普通用户可以注册、登录、退出。
+ 管理员可以用 Milkdown Crepe 写 Markdown 文章。
+ 文章可以设置标题、slug、封面、摘要、分类、标签、发布时间、状态和可见性。
+ 前台可以按首页、分类、标签、归档浏览文章。
+ 未登录访客和普通用户只能看到公开文章。
+ 管理员登录后在前台可以看到全部文章，并能识别草稿、归档、仅自己可见状态。
+ 视觉气质继续保持安静、文学、私人、杂志感。
+ 数据模型不再依赖 Travel / Essay / Journal / Moment 固定枚举。
